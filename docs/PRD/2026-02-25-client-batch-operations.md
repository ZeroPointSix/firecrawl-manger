# PRD：Client 批量管理功能（MVP）

> **创建时间**：2026-02-25
> **状态**：Draft
> **优先级**：P2（功能增强/管理效率）
> **影响范围**：前端 WebUI（Vue 3）+ 后端控制面 API

---

## 1. 背景与问题

### 1.1 现状

- **Key 管理**：已支持批量操作（批量启用/禁用、批量删除等）
- **Client 管理**：仅支持单个操作，缺少批量管理能力
- **实际需求**：管理员在日常运维中需要批量处理 5-10 个 Client

### 1.2 问题

当某些 Client 不再需要时（如测试环境清理、客户端下线等场景），管理员需要：
- 逐个点击启用/禁用按钮
- 逐个确认删除操作
- 操作效率低，容易出错

### 1.3 用户反馈

> "我们的 Key 有批量操作，但是 Client 还是没有批量操作，每次要处理多个 Client 很麻烦。"

---

## 2. 目标（Goals）

- **G1：提升管理效率**：支持一次性操作 5-10 个 Client，减少重复操作
- **G2：功能对齐**：Client 批量管理能力与 Key 批量管理保持一致
- **G3：安全可控**：危险操作（删除、禁用）需要二次确认，防止误操作

---

## 3. 非目标（Non-goals）

- **不支持批量创建**：Client 创建需要导入 Key 等复杂配置，不适合批量操作
- **不支持批量修改名称/描述**：这些字段具有个性化特征，应在单独编辑中处理
- **不在 MVP 中实现批量修改配置**：`rate_limit`、`quota_limit`、`concurrency_limit` 的批量修改放到后续版本

---

## 4. 方案概述（Approach）

### 4.1 MVP 功能范围（第一版）

#### 4.1.1 UI 改动

1. **复选框列**：
   - 在 Client 列表表格最左侧添加复选框列
   - 支持单选、多选、全选
   - 表头复选框用于全选/取消全选

2. **选择状态提示**：
   - 当有 Client 被选中时，显示"已选择 X 个 Client"
   - 未选择时不显示提示

3. **批量操作按钮区域**：
   - 位置：表格上方，与搜索/筛选区域同行
   - 按钮：
     - "批量启用"（仅当选中的 Client 中有禁用状态时显示）
     - "批量禁用"（仅当选中的 Client 中有启用状态时显示）
     - "批量删除"（始终显示）
   - 按钮状态：未选择任何 Client 时，按钮置灰不可点击

#### 4.1.2 功能实现

1. **批量启用**：
   - 将选中的所有 Client 的 `is_active` 设置为 `true`
   - 操作前无需二次确认（启用是安全操作）
   - 操作完成后显示成功提示："已启用 X 个 Client"

2. **批量禁用**：
   - 将选中的所有 Client 的 `is_active` 设置为 `false`
   - 操作前需要二次确认弹窗：
     - 标题："确认批量禁用"
     - 内容："确认禁用 X 个 Client？禁用后这些 Client 将无法访问 API。"
     - 按钮："取消" / "确认禁用"
   - 操作完成后显示成功提示："已禁用 X 个 Client"

3. **批量删除**：
   - 删除选中的所有 Client
   - 操作前需要二次确认弹窗：
     - 标题："确认批量删除"
     - 内容："确认删除 X 个 Client？此操作不可恢复。"
     - 按钮："取消" / "确认删除"（红色危险按钮）
   - 操作完成后显示成功提示："已删除 X 个 Client"

#### 4.1.3 交互规则

1. **操作完成后**：
   - 清空选择状态（所有复选框取消选中）
   - 刷新 Client 列表
   - 显示操作结果提示

2. **部分失败处理**：
   - 如果批量操作中部分成功、部分失败：
     - 显示详细提示："成功 X 个，失败 Y 个"
     - 失败的 Client 保持选中状态，方便重试
     - 在通知中显示失败原因（如果后端返回）

3. **并发操作保护**：
   - 操作进行中，禁用所有批量操作按钮
   - 显示 Loading 状态

### 4.2 后端 API 设计

#### 4.2.1 方案选择

**方案 A：新增批量操作 API**（推荐）
```
PATCH /admin/clients/batch
Body: {
  "client_ids": ["uuid1", "uuid2", ...],
  "action": "enable" | "disable" | "delete"
}
Response: {
  "success_count": 5,
  "failed_count": 0,
  "failed_items": []
}
```

**方案 B：复用现有单个操作 API**
- 前端循环调用 `PATCH /admin/clients/{id}` 或 `DELETE /admin/clients/{id}`
- 优点：无需后端改动
- 缺点：网络开销大，无法保证原子性

**选择**：采用方案 A，提供更好的性能和用户体验。

#### 4.2.2 API 规范

**请求**：
```http
PATCH /admin/clients/batch
Authorization: Bearer {admin_token}
Content-Type: application/json

{
  "client_ids": ["uuid1", "uuid2", "uuid3"],
  "action": "enable" | "disable" | "delete"
}
```

**响应（成功）**：
```json
{
  "success_count": 3,
  "failed_count": 0,
  "failed_items": []
}
```

**响应（部分失败）**：
```json
{
  "success_count": 2,
  "failed_count": 1,
  "failed_items": [
    {
      "client_id": "uuid3",
      "error": "Client not found"
    }
  ]
}
```

**错误响应**：
- `400 Bad Request`：参数错误（如 `client_ids` 为空、`action` 无效）
- `401 Unauthorized`：Admin Token 无效
- `404 Not Found`：所有 Client 都不存在

#### 4.2.3 实现要点

1. **事务处理**：
   - 批量启用/禁用：使用数据库事务，保证原子性
   - 批量删除：使用数据库事务，保证原子性
   - 如果事务失败，回滚所有操作

2. **错误处理**：
   - 记录每个 Client 的操作结果
   - 返回详细的成功/失败统计

3. **审计日志**：
   - 记录批量操作到 `audit_logs` 表
   - 日志内容包含：操作类型、Client ID 列表、操作结果

### 4.3 前端实现要点

1. **状态管理**：
   - 使用 Vue 3 Composition API 管理选择状态
   - 选中的 Client ID 列表存储在组件状态中

2. **API 调用**：
   - 封装批量操作 API 到 `webui/src/api/clients.ts`
   - 统一错误处理和提示

3. **UI 组件**：
   - 使用 Naive UI 的 `n-data-table` 组件（支持复选框）
   - 使用 `n-button` 和 `n-modal` 实现批量操作按钮和确认弹窗

## 6. 风险与对策

### 6.1 部分操作失败

**风险**：批量操作中部分 Client 操作失败（如 Client 不存在、数据库错误等）

**对策**：
- 后端返回详细的成功/失败统计
- 前端显示详细提示，失败的 Client 保持选中状态
- 用户可以重试失败的操作

### 6.2 并发操作冲突

**风险**：多个管理员同时对同一批 Client 进行批量操作

**对策**：
- 使用数据库事务和行锁，保证操作的原子性
- 操作完成后刷新列表，显示最新状态
- 审计日志记录所有操作，便于追溯

### 6.3 误操作风险

**风险**：管理员误选中大量 Client 并执行批量删除

**对策**：
- 批量删除和批量禁用需要二次确认
- 确认弹窗明确显示操作数响
- 审计日志记录所有操作，便于追溯和恢复

### 6.4 性能问题

**风险**：批量操作大量 Client 时，数据库操作耗时过长

**对策**：
- 限制单次批量操作的 Client 数量（如最多 100 个）
- 使用数据库批量更新/删除语句，提升性能
- 操作进行中显示 Loading 状态，避免用户重复点击

## 7. 回滚策略

### 7.1 前端回滚

- 如果前端改动导致 UI 异常：
  - 回滚前端代码到上一个稳定版本
  - 重新构建前端：`cd webui && npm run build`

### 7.2 后端回滚

- 如果后端 API 导致功能异常：
  - 回滚后端代码到上一个稳定版本
  - 重启服务

### 7.3 数据库回滚

- 如果批量操作导致数据异常：
  - 通过审计日志查找异常操作
  - 使用数据库备份恢复数据
  - 或手动修正受影响的 Client 数据
